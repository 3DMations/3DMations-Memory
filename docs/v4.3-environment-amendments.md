### A2. Ollama runs in a shared container, not natively on host

Plan §"Architecture" assumed Ollama ran natively on the host reached via `host.docker.internal:host-gateway`. Actual: Ollama runs in the OPS-owned container `jarvis-ollama` bound to `127.0.0.1:11434` (loopback-only, verified in `3DMations-OPS/docker/inference/docker-compose.yml:13-14`).

**Consequence:** `host.docker.internal:11434` from another container fails — OPS deliberately does not expose Ollama beyond loopback. Verified empirically this session — `curl` from a test container on the default bridge returned "Could not connect to server."

**Fix:** the new hub's `app` service joins the external `jarvis-internal` Docker network and reaches Ollama by container name.

| Variable | Plan value | This environment |
|---|---|---|
| `OLLAMA_URL` | `http://host.docker.internal:11434` | `http://jarvis-ollama:11434` |
| `extra_hosts` | `host.docker.internal:host-gateway` | **Remove** — not needed |
| networks (app) | (default) | Add `jarvis-internal` as external |

Verified reachable: `docker run --rm --network jarvis-internal alpine curl http://jarvis-ollama:11434/api/tags` returns the model list including `qwen3.6:35b`.

### A3. Ollama model — tag that does not exist here

Plan §"Verified Package Versions" used `qwen2.5:32b`. Actual models pulled in `jarvis-ollama` as of 2026-04-24: `qwen3:32b` (20 GB) and `qwen3.6:35b` (23 GB). `qwen2.5:32b` is not present.

OPS-automation (`3DMations-OPS/docker/automation/docker-compose.yml:133-134`) already uses `qwen3.6:35b` — same tag the new hub will use means no duplicate VRAM footprint when loaded.

| Variable | Plan value | This environment |
|---|---|---|
| `OLLAMA_MODEL` | `qwen2.5:32b` | `qwen3.6:35b` |

Memory updated in this session.

### A5. Postgres version — follow plan, PG18

20-row smoke-data audit confirmed the old PG16 data held zero production value, so `pg_upgrade` is moot. Fresh PG18 container gets `uuidv7()` native without workarounds. Image pulled this session (`postgres:18-alpine`, digest `...e1e9d7`).

### A6. Scrubber — client-side per plan, but the old server-side Python scrubber is in `.archive/api/scrubber.py` for reference

Plan §Phase 3 moves scrubber client-side into the Claude Code skill. Doc references to "14 patterns" were drift — `.archive/api/scrubber.py` actually contained 8 patterns. Phase 3b ports those 8 plus 4 modern shapes (Anthropic, Google, SSH/TLS PEM, Tailscale auth keys) for a curated total of 12. See `docs/plan-v4.3-phase-3.md` §3b for the full list.

### A7. Destructive Action Guard — governs this project

CLAUDE.md at project root is authoritative. The plan's `claude -p` execution brief was designed for unattended greenfield runs; we are running interactively with the guard in force. Key implications for Phase 0 and beyond:

- **No `rm`/`rm -rf`.** Archive moves only.
- **No `git reset --hard` / `git push --force` / `git clean`.** Standard git workflow.
- **Any rewrite dropping >5 lines → state + confirm.** No silent truncations.
- **Rule version snapshots** required before any edit to `CLAUDE.md` or `.claude/rules/*.md`, per `memory-system.md:20`.
